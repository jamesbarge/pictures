# Bump Vite to v8 + @sveltejs/vite-plugin-svelte to v7 (frontend)

**PR**: TBD
**Date**: 2026-04-26
**Branch**: `chore/vite-v8`

## Changes

- `vite` 7.3.2 → 8.0.10
- `@sveltejs/vite-plugin-svelte` 6.2.4 → 7.0.0

Both in `frontend/package.json` only. No source code changes. `frontend/vite.config.ts` unchanged.

## Why

Phase 2 item 10 from `tasks/todo.md`. Plugin-svelte's peer-dep requires Vite v8 (`peerDependencies: { vite: "^8.0.0-beta.7 || ^8.0.0" }`), so these two packages MUST move together — pairing them in one PR per the plan.

## The lockfile dance

A naive `npm install vite@^8 @sveltejs/vite-plugin-svelte@^7` failed with ERESOLVE because the existing lockfile retained `@sveltejs/vite-plugin-svelte-inspector@5.0.2` (a transitive of plugin-svelte v6) which expected plugin-svelte v6.

Plugin-svelte v7 dropped the inspector dependency entirely (functionality moved internal — its `dependencies` are now just `obug`, `vitefu`, `deepmerge`, `magic-string`). To clear the stale inspector entry I did:

```bash
cd frontend
rm -rf node_modules package-lock.json
# bump versions in package.json
npm install
```

Clean lockfile, no `--force` or `--legacy-peer-deps` needed.

## What does NOT change

- `frontend/vite.config.ts` — kept the `ssr: { noExternal: ['date-fns'] }` workaround. Tested without it; v8 still fails to externalize `date-fns` cleanly during SSR. Same workaround, same comment.
- `frontend/svelte.config.js` and the rest of the project's Svelte configuration.
